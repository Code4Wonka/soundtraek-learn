Lesson 03 — Storage Model

Schema design, table relationships, and the decisions behind them

What this system does

SoundTræk stores data in seven tables across two categories: global shared reference data (the music catalog) and per-user owned data (runs and listening history). This lesson explains each table, the key design decisions, and how they relate.

Evidence

  • core/models.py
  • core/migrations/0001_initial.py
  • soundtraek/settings.py — database configuration

Two categories of data

Shared tables store the music catalog. If 100 users all listened to the same album, there's one Album row — not 100 copies. Album art color extraction happens once for the whole platform.

User-owned tables store personal activity. Each user's runs and listening events are isolated by user_id.

The music catalog

Album

class Album(models.Model):
    spotify_id = models.CharField(max_length=64, primary_key=True)
    name = models.CharField(max_length=512)
    art_url = models.URLField(max_length=1024, blank=True)
    dominant_color_hex = models.CharField(max_length=7, blank=True)
    palette_json = models.JSONField(default=list, blank=True)
    palette_fetched_at = models.DateTimeField(null=True, blank=True)

Spotify IDs are used as primary keys directly. get_or_create(spotify_id=...) is the natural upsert — no synthetic key needed. palette_fetched_at = NULL means colors haven't been extracted yet.

Track → Album → Artist (M2M)

class Track(models.Model):
    spotify_id = models.CharField(max_length=64, primary_key=True)
    album = models.ForeignKey(Album, on_delete=models.CASCADE, related_name='tracks')
    track_number = models.PositiveSmallIntegerField(default=1)
    assigned_color_hex = models.CharField(max_length=7, blank=True)
    artists = models.ManyToManyField(Artist, related_name='tracks', blank=True)

assigned_color_hex is set after album palette extraction:

assigned_color_hex = palette[track_number % len(palette)]

A 12-track album with 8 palette colors assigns: 0,1,2,3,4,5,6,7,0,1,2,3. No two adjacent tracks share a color, and the whole album stays within its visual family.

Runs and GPS data

Run

class Run(models.Model):
    id = models.UUIDField(primary_key=True, default=uuid.uuid4, editable=False)
    user = models.ForeignKey(User, on_delete=models.CASCADE, related_name='runs')
    source = models.CharField(max_length=16, choices=SOURCE_CHOICES)  # nrc / fit / strava
    external_id = models.CharField(max_length=256)

    class Meta:
        unique_together = [('user', 'source', 'external_id')]

id is a UUID — no integer collision risk when migrating to Supabase. unique_together is the idempotency key: re-running an import against the same files creates no duplicates.

RunPoint

class RunPoint(models.Model):
    run = models.ForeignKey(Run, on_delete=models.CASCADE, related_name='points')
    timestamp = models.DateTimeField()
    distance_m = models.FloatField()
    lat = models.FloatField(null=True, blank=True)    # nullable — not all sources have GPS
    lon = models.FloatField(null=True, blank=True)
    heart_rate = models.PositiveSmallIntegerField(null=True, blank=True)

    class Meta:
        indexes = [models.Index(fields=['run', 'timestamp'])]

Integer PK (not UUID) — high volume, no external identifier needed. Composite index on (run_id, timestamp) makes route queries fast.

Listening history

PlayEvent

class PlayEvent(models.Model):
    id = models.UUIDField(primary_key=True, default=uuid.uuid4, editable=False)
    user = models.ForeignKey(User, on_delete=models.CASCADE, related_name='play_events')
    track = models.ForeignKey(Track, on_delete=models.CASCADE, related_name='play_events')
    played_at = models.DateTimeField(db_index=True)
    source_file = models.CharField(max_length=512, blank=True)

    class Meta:
        unique_together = [('user', 'played_at', 'track')]

The 447 Spotify CSV files overlap at their edges. unique_together on (user, played_at, track) silently discards duplicates during import with no pre-processing required. source_file records the original CSV filename for traceability.

The join table

RunTrackMatch

class RunTrackMatch(models.Model):
    run = models.ForeignKey(Run, on_delete=models.CASCADE, related_name='track_matches')
    play_event = models.ForeignKey(PlayEvent, on_delete=models.CASCADE, related_name='run_matches')
    distance_at_start_m = models.FloatField()
    percent_run = models.FloatField()  # 0.0 to 1.0

This is derived data — it could be recomputed from Run and PlayEvent — but storing it makes the frontend fast. A route map query only needs to join RunPoint → RunTrackMatch → Track.assigned_color_hex with no merge algorithm at request time.

Design rationale

DecisionReason
Color stored on Track, not PlayEventColor is a property of the track (its album). Updating it in one place means no backfilling millions of play event rows.
palette_json as JSON fieldSmall ordered list (≤8 items), always read/written as a whole. No need for a related table.
Nullable sensor fields everywhereDifferent sources have different data quality. Garmin files sometimes have heart_rate = 0 (no HR strap). NRC may lack GPS. Accept and store as-is.
Pre-computed RunTrackMatchThe time-based join is the most expensive operation. Running it once and caching the result keeps all subsequent reads O(1).

Learning takeaway

The schema has one structural insight: split data into what's shared (music catalog) and what's owned (activity data). The join between them (RunTrackMatch) is pre-computed and stored so queries stay simple. UUID primary keys on user-owned records make the eventual Supabase migration straightforward.
