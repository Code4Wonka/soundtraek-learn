Architecture Review

Generated by project-mirror · 2026-03-14

SoundTræk has a well-reasoned foundation. The schema design is solid, the data model is normalized correctly, and the idempotency approach is correct. The main gaps are all expected at this stage — the import pipeline, views, and frontend don't exist yet.

Strengths

Schema design

  • Shared vs user-owned split is correct and will scale cleanly to multi-user
  • UUID PKs on user records make Supabase migration trivial
  • Spotify ID as PK on catalog records is the natural dedup key
  • unique_together constraints make all imports safe to re-run
  • Pre-computing RunTrackMatch keeps frontend queries simple

Color system

  • Palette stored at the album level (shared), not the track level (per-user)
  • Deterministic track_number % palette_length assignment — consistent across runs
  • palette_fetched_at enables incremental, resumable extraction

Project structure

  • Django two-layer separation is correct
  • .env and db.sqlite3 not committed
  • CI in place via GitHub Actions

Gaps and recommendations

GapPriorityNote
Import pipeline not built High Nothing works without data. The four management commands are the critical path.
No views beyond the homepage stub Medium Deferred by design — data layer needs populating first.
No admin registration for models Low Django Admin is a fast way to inspect imported data during development.
AUTH_PASSWORD_VALIDATORS = [] Medium Fine locally. Must be addressed before any login system goes live.
No management command directory High core/management/commands/ doesn't exist yet. Create alongside the first import command.

Not concerns (by design)

  • SQLite in production — Not the plan. Supabase migration is planned and the schema supports it.
  • No caching layer — Premature for a solo project at this stage.
  • No async task queue — Color extraction is a one-off management command. Celery can be added later if live sync is built.
